# XUANOS 前端静态原型开发任务书 v0.1

- 文档版本：v0.1
- 当前阶段：MVP 阶段 1
- 适用范围：XUANOS 静态前端原型
- 开发目标：完成五个核心页面的可运行、可切换静态原型，验证产品气质、信息结构和核心闭环。

---

## 1. 原型目标与边界

本阶段只验证 XUANOS 的前端体验：用户进入系统后，能看懂当前在处理什么、系统如何理解自己、为何做出计划裁决、如何反馈行动结果，以及个人系统如何被持续修正。

原型必须体现以下核心流转：

```text
首页 / 控制台
→ 引导式理解
→ 计划裁决
→ 行动反馈
→ 我的系统
→ 返回首页或继续任务
```

每个页面只解决一个核心问题。原型应呈现“理解、裁决、反馈、修正”的连续体验，而不是五个互不相关的展示页。

本任务不接后端、不接真实 AI、不处理真实用户数据。所有展示内容和交互结果均使用本地 mock 数据模拟。

---

## 2. 技术栈与实现约束

必须使用：

- React
- TypeScript
- Vite
- CSS Modules 或普通 CSS

实现要求：

- 原型可通过 Vite 开发环境在浏览器中启动。
- 页面切换、按钮状态、表单填写和展示状态可由 React 本地状态管理。
- 所有数据来自独立的 mock 数据模块；页面和组件不直接硬编码大段业务数据。
- 暂时不发起真实 API 请求，也不需要服务端、数据库或鉴权。
- 组件的输入数据应使用 TypeScript 类型定义，便于后续将 mock 数据替换为后端 API 响应。
- 业务页面、通用组件、样式和 mock 数据应分离组织，禁止将全部代码堆在 `App.tsx`。

建议的职责划分：

```text
src/
  components/      通用界面组件
  pages/           五个核心页面
  data/            mock 数据与默认展示状态
  types/           前端展示数据类型
  styles/          全局变量、基础样式与页面样式
  App.tsx          路由或页面状态编排，不承载全部页面实现
```

## 3. 全局体验与视觉要求

### 3.1 产品气质

XUANOS 是“个人成长控制台 + 引导型导师入口”，不是普通 AI 聊天网页，不是企业后台，也不是学习打卡 App。

页面应让用户感到：系统正在记录并校准目标、现实条件、行动结果和后续判断；它有温度，但不会回避现实或只提供安慰。

### 3.2 视觉方向

- 黑金东方极简。
- Apple-like 深色毛玻璃，重点用于系统面板、档案面板、计划裁决和首页主卡片。
- 深黑背景，金色仅作为克制的系统强调色、状态色、边框和细分割线。
- 低密度金色粒子作为系统运行感背景：缓慢、低透明度、不遮挡文字；减少动态效果时可弱化或关闭。
- 可少量使用圆相、星图、山水线稿、金色细线等东方极简元素，但不得走向符箓、玄学或国潮插画。
- 文字要有未来感系统气质：中文保持可读，辅以少量英文小标签，例如 `CURRENT VECTOR`、`SYSTEM READ`、`PLAN DECISION`。
- 文案短句化、标签化、状态化。优先呈现结论、阶段、原因、风险、行动和复查条件，避免长篇解释。

### 3.3 信息与布局

- 桌面端采用顶部状态栏、左侧任务栏、中间主工作区、右侧系统摘要的结构。
- 右侧系统摘要固定展示当前主目标、当前唯一行动、系统仍在验证和最近修正；小屏幕可折叠。
- 卡片应体现层级，不得所有内容均使用相同毛玻璃样式。
- 页面可有轻微 hover、切换和毛玻璃高亮，但不做复杂动画、3D 效果或高密度粒子特效。
- 阅读优先：毛玻璃、粒子、细线和状态色均不得影响文字对比度与扫描效率。

### 3.4 明确避免

- 不要普通 AI 聊天页面：首页不能只是消息列表加输入框。
- 不要企业后台风：避免数据表格、密集 KPI、导航堆叠和管理系统式面板。
- 不要学习打卡 App 风：避免连续天数、积分、徽章、排行榜和游戏化激励。

## 4. 五个核心页面

### 4.1 首页 / 控制台 / 对话入口

页面目的：让用户立刻知道当前任务、当前主线和今天唯一要做的事，并选择继续任务、新建目标或进入引导。

必须包含：

- 品牌开场：`XUANOS`、`个人成长操作系统`、`PERSONAL GROWTH OS` 与短句“校准你的下一步。”
- 当前任务线程与“继续上次任务”入口。
- 当前主线卡：主目标、当前阶段、系统状态。
- 当前唯一行动卡：行动、完成标准、预计负荷。
- 三种表达方式：`我先完整表达`、`XUANOS 来问我`、`一起梳理`。
- `ChatComposer` 输入区，placeholder 为“输入你的目标、计划、困境，或让 XUANOS 开始提问。”
- 我的系统入口与导师风格入口。

交互要求：表达方式可选中；“XUANOS 来问我”和“一起梳理”可进入引导式理解页；当前线程、计划和行动反馈入口可切换至对应页面。

首页的重点是“当前主线 + 当前唯一行动 + 启动入口”，聊天输入只能作为一个操作区，不能占据整个页面主体。

### 4.2 引导式理解页

页面目的：让用户感觉系统在逐步理解自己，而非套用模板。

必须包含：

- 页面标题：`引导式理解 / GUIDED UNDERSTANDING`。
- 阶段指示器：理解目标、核对现实、判断冲突、生成起点档案、准备计划；当前阶段金色高亮。
- 系统理解摘要：真实目标、当前基础、现实限制、主要矛盾、仍不确定。
- 下一关键问题卡：一次只呈现一个高价值问题，并可使用单选项或简短输入。
- 用户回答区。
- 纠正入口：`准确`、`部分准确`、`不准确`、`我要补充`；选择“不准确”或“我要补充”时展示简短输入区。
- 继续生成计划按钮。

交互要求：回答、纠正和补充只需更新本地展示状态；完成后进入计划裁决页。不要做长问卷，也不要连续呈现多个问题。

### 4.3 计划裁决页

页面目的：展示 XUANOS 的核心裁决。用户应明显感觉到系统正在帮助自己取舍，而非生成一长串待办。

必须包含：

- 页面标题：`计划裁决 / PLAN DECISION`。
- 主目标卡：主目标、裁决原因、当前阶段、主要风险。
- 目标取舍区：维持目标、暂停目标、删除事项三组内容，视觉层级清晰。
- 当前阶段成果与关键前置或当前瓶颈。
- 时段级计划：高能、普通、低能时段；默认不安排到分钟。
- 当前唯一行动：行动、完成标准、预计负荷，需成为页面最突出元素之一。
- 计划缓冲与复查条件。
- 操作入口：`接受计划`、`修改计划`、`我不同意这个判断`、`重新问我`。
- 修改后的影响提示：普通说明使用金色，可能影响进度或范围使用橙金，事实不可行或安全风险使用暗红。

交互要求：接受计划后可进入行动反馈页；修改计划或不同意判断时，展示 `CorrectionPanel`，对比系统推荐与用户选择，并显示预计影响和复查条件。静态原型中只更新 mock 状态，不需保存版本到后端。

### 4.4 行动反馈页

页面目的：以轻量方式收集真实执行结果，并让用户看见结果如何修正系统判断。

必须包含：

- 页面标题：`行动执行 / ACTION EXECUTION`。
- 原计划摘要与当前唯一行动。
- 是否开始、是否完成、完成比例、实际用时。
- 最大阻力选择：行动不明确、目标不认可、环境干扰、临时意外、他人影响、情绪变化、精力不足、能力不足、其他。
- 情绪或精力变化。
- “原计划哪里不现实”的简短输入。
- 提交反馈按钮。
- 提交后显示 `系统修正 / SYSTEM REVISION`：原判断、实际结果、系统修正、下一步调整。

交互要求：提交前显示表单；提交后切换到 mock 的系统修正结果，并提供进入“我的系统”或返回计划的入口。页面应轻量、短句、易完成，不能做成长篇复盘或复杂情绪量表。

### 4.5 我的系统页

页面目的：让用户看到 XUANOS 正在形成可查看、可纠正、可更新的个人系统，而不是给用户贴固定标签。

必须包含：

- 页面标题：`我的系统 / PERSONAL SYSTEM`。
- `档案模式`、`日记模式`、`混合模式`切换；默认选中混合模式。
- 当前主线：当前主目标、当前阶段、当前唯一行动。
- 现实边界：可用时间、固定安排、身体或资源限制、近期不可改变事项。
- 对我有效：基于行为证据的有效规律。
- 系统仍在验证：明确标记为假设，不得包装成定论或显示伪科学百分比。
- 当前计划、最近修正与用户纠正记录。
- 每条系统信息的操作：`准确`、`部分准确`、`不准确`、`已经变化`、`查看依据`、`不希望继续使用`。

模式要求：

- 档案模式突出结构化信息、现实边界、行为证据和最近修正。
- 日记模式突出最近状态、近期事件、用户表达、系统复述和行动记录。
- 混合模式上半部分展示档案，下半部分展示近期日记记录。

交互要求：模式切换应实际改变布局或信息侧重。用户纠正操作应在前端显示已更新的状态或一条新的纠正记录；不需要真实持久化。

## 5. 组件拆分

以下组件为静态原型的最低拆分要求。可根据实现增加小型辅助组件，但不得合并到单一页面文件或 `App.tsx` 中。

| 组件 | 责任 |
|---|---|
| `AppShell` | 承载全局三栏结构、当前页面和响应式布局。 |
| `Sidebar` | 展示品牌、任务线程和核心页面导航。 |
| `TopStatusBar` | 展示当前线程、导师风格、系统状态与快速入口。 |
| `ParticleBackground` | 提供低密度、低干扰的金色粒子背景。 |
| `GlassPanel` | 深色毛玻璃面板的统一容器。 |
| `ThreadCard` | 展示当前或历史任务线程摘要。 |
| `ModeSelector` | 选择“我先完整表达 / XUANOS 来问我 / 一起梳理”。 |
| `ChatComposer` | 目标、困境或计划的短输入区及快捷操作。 |
| `UnderstandingCard` | 展示系统理解摘要与当前阶段。 |
| `QuestionCard` | 展示唯一关键问题、选项或简短回答区。 |
| `PlanDecisionCard` | 展示主目标、裁决原因、阶段与风险。 |
| `GoalPriorityCard` | 展示维持、暂停或删除目标的取舍信息。 |
| `TimeBlockPlan` | 展示高能、普通、低能时段建议与缓冲。 |
| `ActionCard` | 强调当前唯一行动、完成标准和负荷。 |
| `FeedbackForm` | 收集开始、完成、比例、用时、阻力与现实偏差。 |
| `SystemSnapshotCard` | 展示个人系统的单个摘要板块。 |
| `CorrectionPanel` | 展示用户纠正、系统推荐、影响与复查条件。 |
| `WarningBanner` | 展示 Info、Impact、Risk 等不同级别提示。 |
| `PrimaryButton` | 关键推进操作，例如生成计划、接受计划、提交反馈。 |
| `SecondaryButton` | 次级操作，例如修改、补充、查看依据。 |
| `Tag` | 展示短标签、阶段、负荷、状态和提示等级。 |
| `SystemViewToggle` | 切换档案、日记和混合模式。 |

## 6. Mock 数据规范

所有页面共用同一份演示数据，默认用户为 `demo-user`。至少包含以下内容：

```text
当前任务：XUANOS 暑假开发
主目标：完成 XUANOS 静态前端原型
当前阶段：视觉系统确认
当前唯一行动：完成五个页面线框
维持目标：每周 3 次基础健身
暂停目标：Flutter 客户端、完整商业系统
系统仍在验证：用户可能通过继续完善文档推迟真实开发
我的系统模式：混合模式
```

Mock 数据还应覆盖下列展示状态：

- 当前任务线程：状态为“进行中”或“等待行动”，可继续上次任务。
- 导师风格：默认“平衡分析”，并显示它是可调整的初始偏好。
- 理解摘要：真实目标、当前基础、现实限制、主要矛盾、仍不确定。
- 单个关键问题及其选项，例如“完成五个页面线框后，下一步最应该验证什么？”
- 计划裁决：主目标、维持目标、暂停目标、删除事项、时段建议、唯一行动、完成标准、复查条件与主要风险。
- 修改计划示例：系统推荐方案、用户选择、预计影响、复查条件。
- 行动反馈示例：是否开始、完成比例、实际用时、最大阻力、精力变化与原计划偏差。
- 系统修正示例：原判断、实际结果、新判断、下一步调整。
- 个人系统摘要：当前主线、现实边界、对我有效、待验证假设、当前计划、最近修正和用户纠正。

数据语义要求：

- 用户确认的事实、用户表达、系统假设和行为证据应在文案或标签上区分。
- “系统仍在验证”只作为待验证假设展示，需附带“当前证据不足”或“等待下一次行动结果验证”等提示。
- 计划应是时段级而非分钟级，并保留缓冲和复查条件。
- 计划修改不应表现为静默覆盖：原型中至少呈现“系统推荐”和“用户最终选择”的差异。

## 7. 明确禁止事项

- 不要接真实 AI。
- 不要做登录注册。
- 不要做数据库。
- 不要做支付。
- 不要做复杂动画。
- 不要所有代码堆在 `App.tsx`。
- 不要新增 MVP 范围外页面。
- 不要新增社区、排行榜、积分、课程体系、移动端 App 或完整商业系统入口。
- 不要用精确成功概率、人格诊断或未经证实的结论包装系统判断。
- 不要将首页实现成单纯的聊天消息列表。

## 8. 验收标准

静态原型完成时，必须满足：

1. 页面能在浏览器跑起来。
2. 五个核心页面能切换，并能呈现首页 → 理解 → 裁决 → 反馈 → 我的系统的主流程。
3. 首页不像普通聊天框，能优先看到当前主线、当前唯一行动和启动方式。
4. 计划页具有明确裁决感，能清楚呈现主目标、维持目标、暂停目标、删除事项和当前唯一行动。
5. 我的系统页有档案 / 日记 / 混合模式切换，且切换后信息呈现实际变化。
6. 粒子和毛玻璃不影响阅读、滚动和信息层级。
7. 所有数据暂时来自 mock，不依赖真实 AI、后端接口或数据库。
8. 页面与组件使用明确的 TypeScript 数据类型和独立 mock 数据源，后续可以方便接入后端 API。
9. 引导页一次只展示一个关键问题，用户可确认、纠正或补充系统理解。
10. 反馈提交后可看到系统修正的 mock 结果。
11. 系统假设与用户事实有清晰区分，且用户可在“我的系统”中纠正或停用信息。
12. 视觉整体符合黑金东方极简、深色 Apple-like 毛玻璃、低密度粒子和系统化短文案要求。

## 9. 交付说明

本阶段的交付物是前端静态原型，不是可上线产品。重点是验证视觉方向、页面信息层级、核心交互状态和产品闭环表达是否成立。

在原型通过验收后，再按既定顺序进入后端 health 接口、数据模型、任务线程接口、understanding mock、plan mock、feedback 与 snapshot 的开发与联调。
