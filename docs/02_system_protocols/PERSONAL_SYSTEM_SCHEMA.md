# XUANOS 个人系统数据结构 v0.1
## PERSONAL_SYSTEM_SCHEMA.md

- 文档版本：v0.1
- 当前状态：产品规格草案
- 适用版本：XUANOS MVP
- 核心目标：定义系统应长期保存、短期保存、展示给用户和仅供后台判断的信息
- 核心原则：不建立固定“用户画像”，而维护一套可纠正、可过期、有证据、有版本的动态个人系统

---

# 1. 文档目的

本文件规定 XUANOS 的个人系统由哪些数据构成、这些数据从哪里获得、如何确认、何时过期、如何纠正，以及如何影响后续计划与对话。

个人系统不是聊天记录的堆积，也不是一次性测评结果。

它必须能够回答：

1. 用户当前真正想推进什么；
2. 用户拥有和缺少什么条件；
3. 当前有哪些硬性约束；
4. 用户最近处于什么状态；
5. 过去实际执行了什么；
6. 哪些系统判断已经被行为支持；
7. 哪些判断仍只是待验证假设；
8. 用户曾纠正过系统什么；
9. 哪种导师方式更适合当前用户；
10. 下一次对话时，系统应该基于什么继续工作。

---

# 2. 三层数据架构

## 2.1 原始事件层 Event Log

保存“发生过什么”，不急着解释。

包括：用户说了什么、系统问了什么、目标与计划如何变化、用户是否执行、系统为什么修改判断。

规则：

- 原始事件原则上只追加，不覆盖；
- 每条事件带时间、来源、任务线程和关联对象；
- 删除长期模型中的某条结论，不等于删除对应历史事件；
- 用户要求删除的非必要信息，应从可用模型中移除，并按隐私规则处理。

## 2.2 结构化模型层 Structured Model

将原始信息整理成目标、约束、状态、计划、行为、假设、规律等结构化对象。

必须区分：用户事实、用户表达、系统假设、行为证据、已验证规律、用户纠正、当前有效信息、已过期信息。

## 2.3 当前快照层 Current Snapshot

每次对话优先读取当前快照，而不是直接读取全部历史。

当前快照包含：

- 当前主目标；
- 当前维持目标；
- 当前暂停目标；
- 本周关键约束；
- 当前状态；
- 最近行动结果；
- 已验证个人规律；
- 待验证关键假设；
- 导师风格；
- 最近用户纠正；
- 当前唯一行动；
- 需要下一轮确认的信息。

快照在目标、约束、状态、计划、行动结果、假设、纠正或信息有效期发生变化时更新。

# 3. 信息可信度等级

## L1 用户表达

用户当时的主观说法，例如“我就是不自律”。不能直接视为事实。

## L2 用户确认事实

用户明确提供、相对客观的信息，例如固定课程、截止日期、当前技术基础。

## L3 系统假设

系统根据表达、事实和历史行为提出的暂时解释。必须带证据和置信度，不得包装成事实。

## L4 单次行为证据

一次真实行动结果，例如任务缩小后当天完成。

## L5 稳定个人规律

至少在两次不同情境中得到支持，且没有强反证。首版两次形成“候选规律”，三次以上形成“稳定规律”。

# 4. 核心实体

## 4.1 users

账户与基础设置。

| 字段 | 必填 | 来源 | 用户可见 | 说明 |
|---|---:|---|---:|---|
| id | 是 | 系统 | 否 | 用户唯一标识 |
| display_name | 否 | 用户 | 是 | 显示名称 |
| age_range | 否 | 用户 | 是 | 年龄段，不强制精确生日 |
| student_stage | 否 | 用户 | 是 | 初中/高中/大学/其他 |
| timezone | 否 | 系统或用户 | 是 | 用于时段和提醒 |
| consent_version | 是 | 系统 | 是 | 数据规则版本 |
| created_at | 是 | 系统 | 否 | 创建时间 |
| updated_at | 是 | 系统 | 否 | 更新时间 |

禁止默认保存精确住址、无关家庭隐私、医疗诊断和无关敏感信息。

## 4.2 mentor_preferences

- directness：温和/平衡/直接；
- explanation_depth：简洁/标准/深入；
- decision_style：提供选择/明确建议；
- challenge_level：低/中/高；
- emotion_response：少/适中/多；
- user_selected：是否由用户主动选择；
- system_adjustment：系统观察后提出的适配建议；
- last_confirmed_at：最近确认时间。

系统可以适应，但不得静默彻底改变导师风格。

## 4.3 goals

| 字段 | 必填 | 说明 |
|---|---:|---|
| original_expression | 是 | 用户原话 |
| title | 是 | 澄清后的目标名称 |
| goal_type | 是 | 学习/项目/习惯/考试/身体/其他 |
| desired_outcome | 是 | 最终想得到什么 |
| success_criteria | 是 | 怎样算完成 |
| purpose | 否 | 为什么想完成 |
| source | 否 | 主动/外部/混合 |
| deadline | 否 | 截止日期 |
| priority | 是 | 主目标/维持/暂停/放弃 |
| status | 是 | 待确认/进行中/暂停/完成/取消 |
| current_stage | 否 | 当前阶段 |
| prerequisite_ids | 否 | 前置目标或任务 |
| conflict_goal_ids | 否 | 冲突目标 |
| estimated_load | 否 | 低/中/高/过载 |
| feasibility | 否 | 低/中/高 |
| feasibility_basis | 否 | 判断依据 |
| user_confirmed | 是 | 是否确认 |
| last_reviewed_at | 是 | 最近复查 |
| valid_until | 否 | 适用期限 |

目标原话与澄清后目标必须同时保留。

## 4.4 constraints

约束类型：

1. safety：安全约束；
2. fixed：固定约束；
3. resource：资源约束；
4. ability：能力约束；
5. preference：个人偏好；
6. temporary_event：临时事件。

每条约束保存：内容、类型、严重度、来源、是否硬性、生效时间、结束时间、用户是否确认、影响哪些目标、证据和最近复核时间。

“我想休息”不得自动归为偏好，需要确认是恢复需求、疲惫、身体问题还是单纯选择。

## 4.5 user_states

首版状态：稳定、迷茫、焦虑、挫败、抵触、兴奋或目标膨胀、疲惫、注意力分散。

每条状态保存：类型、强度、来源、置信度、对决策的影响、记录时间、过期时间、用户是否确认。

短期情绪不得自动写成长期人格特征。

## 4.6 plans

计划必须有版本。

保存：目标、版本号、状态、是否系统推荐、是否用户选择的非首选方案、计划摘要、可行性、负荷、提示等级、提示文案、复查条件和创建时间。

计划不得直接覆盖。每次修改生成新版本。

## 4.7 plan_items

保存：任务名称、任务类型、时段、具体时间（可空）、预计用时、难度、重要性、前置任务、完成标准、是否可选、来源、是否用户修改和修改说明。

## 4.8 action_results

保存：是否开始、是否完成、完成比例、原定开始、实际开始、实际耗时、主观难度、主要阻力、阻力描述、情绪变化、临时事件、用户复盘和提交时间。

主要阻力：行动、目标、环境、意外、人物、情绪、精力、能力、未知。

## 4.9 hypotheses

保存：假设内容、类别、内部置信度、状态、支持证据、反对证据、是否需用户确认、用户态度、创建时间和最近复核。

状态：待验证、得到支持、被削弱、被推翻、已过期。

前台不得显示伪科学百分比。

## 4.10 personal_rules

保存：规律内容、类别、成熟度、支持次数、反证次数、内部置信度、前台简要表达、用户是否认可、创建时间、最近验证和有效期。

## 4.11 user_corrections

保存：被纠正对象、旧内容、用户修改、原因、系统处理、是否仍有矛盾和时间。

## 4.12 user_events

事件类型包括：用户消息、系统追问、理解复述、用户确认、用户纠正、目标创建、目标更新、计划创建、计划修改、任务开始、结果提交、假设更新、规律形成、状态变化。

每条事件至少包含：用户、任务线程、事件类型、数据内容、时间、来源和关联对象。

# 5. 信息写入规则

## 可直接写入

- 用户确认的目标；
- 截止日期；
- 固定课程和不可用时间；
- 用户选择的导师风格；
- 计划修改；
- 真实执行结果；
- 用户明确纠正。

## 需确认后写入

- 目标真实动机；
- 伪努力判断；
- 逃避模式；
- 某种导师风格更有效；
- 长期能力判断；
- 稳定个人规律。

## 仅短期保存

- 当天情绪；
- 当前疲惫；
- 临时矛盾；
- 短期兴奋；
- 一次性意外。

## 不保存

- 无关敏感隐私；
- 没有产品价值的闲聊细节；
- 未经证实的心理诊断；
- 用户要求删除的非必要信息；
- 系统无法说明用途的信息。

---

# 6. 前台“我的系统”

用户可见六个板块：

1. 当前主线；
2. 现实边界；
3. 对我有效；
4. 系统仍在验证；
5. 当前计划；
6. 最近变化。

每项提供：准确、部分准确、不准确、已经变化、查看依据、不希望继续使用。

后台完整数据不全部公开，但用户必须能够理解影响其计划的关键内容。

---

# 7. 快照生成规则

每次会话结束时生成：当前主目标、维持目标、暂停目标、有效约束、当前状态、最近行动结果、稳定规律、待验证假设、导师风格、最近纠正、当前行动和缺失信息。

快照只保留当前决策最相关的信息。

---

# 8. v0.1 最小实现

首版必须实现：用户、导师风格、任务线程、目标、约束、当前状态、计划版本、计划任务、行动结果、系统假设、用户纠正和当前快照。

首版暂缓：群体统计、精确完成概率、自动读取手机行为、复杂规律图谱、多人共享和完整心理模型。

---

# 9. 验收标准

1. 用户第二次进入时无需重复解释全部背景；
2. 计划修改不会覆盖历史版本；
3. 情绪信息会过期；
4. 系统假设与用户事实明确区分；
5. 用户可以纠正系统；
6. 行动结果能够改变后续计划；
7. 个人规律能追溯到证据；
8. 用户可以查看简略个人系统；
9. 系统能说明建议基于哪些数据；
10. 无关敏感信息不会进入长期模型。
